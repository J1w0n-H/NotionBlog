## 총평

이 사이트는 초급 개발자가 AI의 도움을 받아 빠르게 제작한 결과물입니다.
겉으로 보기엔 완성도가 높고 실제로 잘 돌아갑니다.
그러나 코드 내부를 살펴보면 **"지금 당장 문제는 없지만 곧 터질 것들"**이 여럿 있고,
**"이미 조용히 오작동하고 있는 것"**도 존재합니다.

## 1. 기능 문제

> 사용자가 직접 겪거나, 데이터 손실·보안으로 이어지는 문제

---

### 🔴 F-1. 포스트 101번째부터 사이트에 표시되지 않음

**무슨 일이 일어나고 있나**

Notion에서 글을 100편 초과로 쓰면 101번째 글부터 사이트에 나타나지 않습니다.
에러 메시지도, 경고도 없이 그냥 사라집니다.

**왜 이런 일이 생기나**

Notion API는 한 번에 최대 100개의 데이터를 반환합니다.
100개가 넘으면 "다음 페이지"를 추가로 요청해야 하는데,
현재 코드는 첫 번째 요청만 하고 끝냅니다.

```
현재: Notion에 글 150편 → 사이트에 100편만 표시 → 나머지 50편 소리 없이 누락
정상: Notion에 글 150편 → "다음 페이지" 재요청 → 사이트에 150편 모두 표시
```

**이미 프로젝트 안에 올바른 구현이 있습니다**
`queryAllDatabasePages.ts`라는 파일에 "다음 페이지 자동 요청" 로직이 이미 완성되어 있으나,
실제 글 목록을 가져오는 코드에서 이걸 사용하지 않고 있습니다.
수정 난이도: **낮음** (파일 하나 수정)

---

### 🔴 F-2. 번역 API — 아무나 무제한 호출 가능

**무슨 일이 일어나고 있나**

사이트에 `/api/translate` 라는 번역 서비스가 있습니다.
이 서비스는 비밀번호도, 횟수 제한도 없이 완전히 열려 있습니다.
악의적인 사람이 이 주소를 발견하면 반복 호출로 외부 번역 API(Google 등)를 남용할 수 있습니다.
결과적으로 번역 서비스가 차단되어 **사이트 번역 기능 전체가 멈출 수 있습니다.**

**영향:**
- 서비스 중단 위험 (번역 기능)
- 외부 무료 API의 IP 차단 → 번역 불능 상태

**수정 방향:** 시간당 호출 횟수 제한(Rate Limit) 추가. 난이도: **낮음**

---

### 🟠 F-3. 번역 실패 시 아무 알림 없음

**무슨 일이 일어나고 있나**

번역 버튼을 누르면 Google → MyMemory → Lingva 순서로 세 군데 외부 서비스에 번역을 요청합니다.
세 곳 모두 실패하면 번역이 안 되는데, 사용자에게는 아무 메시지도 표시되지 않습니다.
그냥 번역 패널이 조용히 비어있게 됩니다.

**영향:** 사용자가 번역 기능이 망가진 건지, 아직 로딩 중인 건지 알 수 없음

---

### 🟠 F-4. 글 이미지(썸네일)가 일정 시간 후 깨질 수 있음

**무슨 일이 일어나고 있나**

Notion 내부에 업로드된 이미지는 임시 URL로 제공됩니다.
이 URL은 **약 1시간 후 만료**됩니다.
사이트가 Notion에서 이미지 주소를 가져온 뒤, 방문자가 1시간 이상 지나서 같은 페이지를 보면
썸네일이나 글 속 이미지가 깨진 아이콘으로 표시될 수 있습니다.

**수정 방향:** Notion 이미지를 별도 CDN에 복사하거나, 항상 직접 Notion URL을 재요청하도록 변경

---

### 🟠 F-5. 검색이 현재 화면에 보이는 글만 대상

**무슨 일이 일어나고 있나**

검색창에 단어를 입력하면 **글 제목과 요약문**만 검색합니다.
글 본문 내용은 검색 대상이 아닙니다.
또한 현재 불러온 글 목록에서만 검색하므로, 위 F-1 문제(100개 제한)와 겹치면
검색 가능한 글 자체가 누락될 수 있습니다.

---

### 🟡 F-6. 글 하나 생성에 최대 5분 허용

**무슨 일이 일어나고 있나**

사이트를 빌드(새로 배포)할 때, 각 블로그 글의 Notion 내용을 가져오는 데 시간이 오래 걸립니다.
이를 해결하기 위해 "최대 5분까지 기다리도록" 설정이 되어 있습니다.
이는 임시방편이며, 글이 많아질수록 전체 빌드 시간이 기하급수적으로 늘어납니다.

코드 내 주석에도 "나중에 제대로 고쳐야 함"이라고 적혀 있습니다.

---

### 🟡 F-7. 댓글 시스템이 두 개 동시에 켜져 있음

**무슨 일이 일어나고 있나**

Cusdis와 Utterances, 두 가지 댓글 서비스가 모두 "활성화" 상태입니다.
실제로는 Cusdis 하나만 작동하고 Utterances는 무시됩니다.
사용자 입장에서는 문제없어 보이지만, 설정이 혼란스럽고 나중에 댓글 서비스를 바꿀 때
오해의 소지가 생깁니다.

또한 Cusdis의 서비스 인증 키(`appid`)가 설정 파일에 평문으로 노출되어 있습니다.

## 2. 코드 문제

> 지금 당장 눈에 보이지는 않지만, 유지보수를 어렵게 하거나 나중에 기능 추가를 방해하는 문제

---

### 🔴 C-1. 같은 코드가 두 군데 따로 존재 (글 목록 처리 로직)

**비유로 설명하면**

요리법이 두 권의 레시피북에 각각 실려 있는데, 두 버전이 조금씩 다릅니다.
나중에 레시피를 수정할 때 두 군데 모두 고쳐야 하는데, 한 곳만 고치면 다른 쪽이 오래된 상태가 됩니다.

**실제 상황**

Notion에서 글 데이터를 꺼내는 코드가 두 파일에 각각 구현되어 있습니다.

- `getPosts.ts` — 타입 검사 없이 급하게 짠 버전 (타입 `any` 범람)
- `mapNotionDatabasePage.ts` — 제대로 타입을 갖춘 버전

그런데 `mapNotionDatabasePage.ts`를 만들어놓고 정작 `getPosts.ts`는 이걸 사용하지 않고
자체 로직을 따로 실행합니다.

**결과:**
- 두 버전 간에 처리 방식이 미묘하게 달라 예기치 않은 동작 발생 가능
- 둘 중 하나를 고쳐도 다른 하나에 반영 안 됨

---

### 🔴 C-2. 타입 안전성 없음 (`any` 타입 남발)

**비유로 설명하면**

창고 물건에 "뭔지 모르는 박스"라고 라벨을 붙이는 것입니다.
나중에 꺼낼 때 뭐가 들어있는지 아무도 모릅니다.

**실제 상황**

`getPosts.ts` 파일 전체에서 모든 Notion 데이터를 "어떤 형태든 다 받겠다(`any`)"고 선언해 놓았습니다.
이러면 데이터가 예상과 다른 형태로 들어왔을 때 에러를 미리 잡지 못하고
실제 사용자 화면에서 터지게 됩니다.

**관련 코드 4곳:**
```
(page: any), (prop: any): any, convertedProps: any, (a: any, b: any)
```

---

### 🟠 C-3. 개발 과정 산출물 파일이 레포에 그대로

**무슨 일이 일어나고 있나**

개발 중에 임시로 만든 문서와 파일들이 정리되지 않은 채 남아 있습니다.

| 파일명 | 용도 | 문제 |
|--------|------|------|
| `NOTION_SDK_MIGRATION.md` | 개발 과정 기록 | 완료된 작업 문서. 관리자 외 노출 불필요 |
| `notion_relabel_plan.csv` | 작업 계획표 | 스프레드시트. 레포에 있을 이유 없음 |
| 루트의 `profile.jpg` | 사용처 불명 | `public/profile.jpg`와 중복 의심 |

프로젝트 루트가 지저분해지고, 소스코드를 공개할 경우 내부 작업 계획이 노출됩니다.

---

### 🟠 C-4. 설정 파일 이중 관리

**비유로 설명하면**

같은 계약서가 두 권 있는데, 한 권은 실제 내용이고 다른 한 권은 목차만 있습니다.
둘이 항상 일치해야 하는데 누가 관리해야 하는지 불분명합니다.

**실제 상황**

- `site.config.js` — 실제 설정값이 들어있는 파일 (JavaScript)
- `site.config.d.ts` — 그 파일의 "타입 설명서" (TypeScript)

이 두 파일을 항상 동기화해야 합니다. 하나를 바꾸면 다른 하나도 바꿔야 하는데,
까먹으면 타입 오류가 발생하거나 잘못된 타입 정보를 믿게 됩니다.

**수정 방향:** 두 파일을 `site.config.ts` 하나로 통합

---

### 🟠 C-5. 한 파일이 780줄 — 너무 큰 컴포넌트

**비유로 설명하면**

설계도, 재료 목록, 공사 일정이 한 장의 종이에 다 적혀있는 것입니다.
수정할 때 어디를 봐야 하는지 찾기가 어렵습니다.

**실제 상황**

`SectionNav.tsx` (좌측 네비게이션 바) 하나가 780줄입니다.
카테고리 계산, 태그 필터, 검색 입력, UI 렌더링, 스타일까지 전부 한 파일에 있습니다.
수정하거나 기능을 추가할 때 의도치 않은 곳을 건드릴 위험이 높습니다.

---

### 🟡 C-6. About 텍스트 전체가 코드 안에 하드코딩

**무슨 일이 일어나고 있나**

About 페이지의 모든 텍스트(영어+한국어)가 `aboutContent.ts` 파일에 코드로 박혀있습니다.
단어 하나를 수정하려면 코드를 편집하고 → 저장하고 → 빌드하고 → 배포해야 합니다.
일반 텍스트 편집과 달리, 매 수정마다 개발자가 필요합니다.

**블로그 글은 Notion에서 자유롭게 편집 가능한데, 정작 About 페이지는 코드 수정만 가능**한
구조적 모순이 있습니다.

---

### 🟡 C-7. 한국어 번역 딕셔너리가 코드에 하드코딩

**무슨 일이 일어나고 있나**

카테고리 이름의 한국어 번역이 `i18n.ts` 파일에 직접 적혀 있습니다.
Notion에서 카테고리 이름을 변경하면 → 코드의 번역 딕셔너리도 직접 수정해야 합니다.
이 둘이 어긋나면 한국어 모드에서 카테고리 이름이 영어로 표시됩니다.

---

### 🟡 C-8. 동일한 라이브러리를 두 번 불러옴

**실제 상황**

코드 하이라이팅 라이브러리(`prism-go.js`)를 한 파일에서 두 번 import합니다.

```javascript
// RootLayout/index.tsx 24번째 줄
import 'prismjs/components/prism-go.js'
// ...
// 44번째 줄 — 동일한 import 중복
import "prismjs/components/prism-go.js"
```

기능적 오류는 없지만, 코드 리뷰나 자동 검사 도구에서 지적받는 품질 문제입니다.

---

### 🟡 C-9. 미완료 작업 메모(TODO)가 코드에 남아 있음

`getPosts.ts` 파일 상단에:
```
// TODO: react query를 사용해서 처음 불러온 뒤로는 해당데이터만 사용하도록 수정
```

이미 React Query가 도입되어 있는데 이 TODO가 정리되지 않고 남아있습니다.
완료 여부가 불분명하고, 새 개발자가 이 코드를 보면 혼란스럽습니다.

## 3. 디자인 문제

> 시각적 완성도와 사용자 경험에 영향을 미치는 문제

---

### 🔴 D-1. 상단 배너의 GitHub 링크가 잘못됨

**무슨 일이 일어나고 있나**

홈 화면 상단의 보라색 배너에 "view source ↗" 링크가 있습니다.
이 링크가 가리키는 GitHub 주소가 틀려있습니다.

```
현재 링크: github.com/jiwon-lieb/notion-blog  ← 존재하지 않는 주소
올바른 주소: github.com/j1w0n-h/...
```

방문자(특히 채용 담당자)가 클릭하면 404 페이지로 이동합니다.
포트폴리오 사이트에서 소스코드 링크가 깨져있다는 건 인상이 나쁩니다.

**수정 난이도:** 매우 낮음 (링크 주소 1개 수정)

---

### 🔴 D-2. 라이트 모드에서 카테고리 색상 구분이 사라짐

**무슨 일이 일어나고 있나**

이 사이트는 8가지 카테고리마다 다른 색상을 배정해서 시각적으로 구분합니다.
이 기능이 **다크 모드에서만 제대로 작동**합니다.

OS를 라이트 모드로 설정한 방문자가 사이트를 방문하면,
모든 카테고리 태그가 **같은 보라색 하나**로 표시됩니다.
카테고리 구분 기능이 사실상 비활성화되는 것과 같습니다.

**원인:** 라이트 모드 OS 매칭 CSS에서 카테고리별 색상값 대신 보라 단일 색상을 반복 지정

```css
/* sentinel-theme.css — OS 다크모드 감지 시 */
--cat-reverse-solid: #9b6cff;   ← 전부 동일한 보라
--cat-systems-solid: #9b6cff;   ←
--cat-ctf-solid:     #9b6cff;   ←
/* ... 8개 카테고리 모두 동일 */
```

---

### 🟠 D-3. 모바일에서 경력 설명 툴팁을 볼 수 없음

**무슨 일이 일어나고 있나**

홈 화면의 "Background" 섹션(타임라인)에서 키워드 태그에 마우스를 올리면
상세 설명이 팝업으로 나타납니다.

```
데스크톱: [Cloud Migration] → 마우스 오버 → "기업 이전 및 M365 마이그레이션..." 표시
모바일:   [Cloud Migration] → 탭(터치) → 아무것도 안 나옴
```

모바일 사용자는 이 기능이 아예 없는 것과 같습니다.
스마트폰으로 사이트를 보는 방문자(특히 채용 담당자)에게 경력 상세를 전달하지 못합니다.

**수정 방향:** 모바일에서 탭하면 툴팁이 켜지고 다시 탭하면 닫히도록 토글 처리

---

### 🟠 D-4. 프로필 카드 텍스트가 코드에 하드코딩

**무슨 일이 일어나고 있나**

홈 화면 상단 프로필 카드의 두 가지 핵심 요소가 코드에 직접 박혀있습니다.

**1) 설명 문구 (하드코딩):**
```
"Ran a 200-node cluster for four years, then came to Maryland to learn
```

### 🟠 D-6. 테마 색상 변수를 만들었는데 컴포넌트에서 직접 색상값 사용

**비유로 설명하면**

회사 브랜드 컬러 가이드북을 만들었는데, 디자이너들이 직접 팬톤 번호를 구글에서 찾아서 씁니다.
나중에 브랜드 색상을 변경하면 가이드북만 바꿔선 안 되고 모든 디자이너에게 개별 통보해야 합니다.

**실제 상황**

CSS 변수 시스템(`--accent`, `--link` 등)을 만들어서 색상을 체계적으로 관리하려 했는데,
실제 컴포넌트 코드 **92군데**에서 이 변수를 쓰지 않고 원시 색상값을 직접 씁니다.

```css
/* 이렇게 써야 하는데 */
color: var(--accent);

/* 이렇게 직접 씀 */
color: rgba(155,108,255,.4);
```

브랜드 색상을 바꾸고 싶을 때 92군데를 하나하나 찾아서 바꿔야 합니다.

### 🟡 D-7. 라이트 모드 완성도 미흡

**무슨 일이 일어나고 있나**

설정에서 라이트/다크/시스템 테마를 선택할 수 있는 기능이 있습니다.
그러나 이 사이트는 처음부터 다크 모드를 중심으로 디자인되었고,
일부 컴포넌트에 다크 모드의 구체적인 색상값(`rgba(8,6,17,...)`)이 하드코딩되어 있어
라이트 모드로 전환하면 일부 요소가 어색하게 보입니다.

예시: 글 패널 우측 배경 (`rgba(14, 11, 26, 0.62)`) — 이 색은 다크 배경에서만 자연스럽습니다.
